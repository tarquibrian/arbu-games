# Guía de Materiales Derivados del Documento Base

**Referencia rápida para crear y usar los 7 documentos que nacen del documento base "Arbu Games - 2026"**

---

## Descripción General

El documento base "Arbu Games - 2026" es un documento **vivo y estratégico** que sirve como fuente única de verdad. A partir de él, se derivan 7 materiales especializados, cada uno con propósito, audiencia y formato distintos.

```
DOCUMENTO BASE
├── Documento Estratégico para Socios y Aliados
├── PDF Institucional
├── Documento DOCX Editable
├── Pitch Deck (Comercios, Gobiernos, Inversionistas)
├── Business Model Canvas
├── Roadmap de Producto
└── Plan de Validación del MVP
```

---

## 1. Documento Estratégico para Socios y Aliados

### Descripción
Versión simplificada y enfocada del documento base, adaptada para compartir con organizaciones que podrían ser socios o aliados (universidades, ONGs, municipios, inversionistas).

### Audiencia
- Posibles socios estratégicos
- Inversionistas potenciales
- Instituciones públicas (municipios, gobiernos)
- Organizaciones ambientales

### Cuándo Usarlo
- En primeras conversaciones con aliados potenciales
- Antes de reuniones de partnership
- Cuando necesitas "pitch de 5 minutos" por escrito

### Diferencias con el Base
| Aspecto | Base | Estratégico |
|---------|------|-------------|
| Extensión | 15+ páginas | 3-5 páginas |
| Enfoque | Exhaustivo, conceptual | Impacto, oportunidad |
| Lenguaje | General | Personalizado por aliado |
| Detalles | Todos | Solo los relevantes |
| Objetivo | Entender el proyecto | Motivar colaboración |

### Estructura Sugerida
```
1. Portada (nombre aliado)
2. Resumen ejecutivo (1 página)
3. El problema y la solución (1 página)
4. Valor específico para ESTE aliado (1-2 páginas)
5. Propuesta concreta (1 página)
6. Próximos pasos
```

### Ejemplo de Valor Específico
**Para Universidad:** "Acceso a datos para investigación + base científica + publicaciones conjuntas"  
**Para Municipio:** "Inventario forestal dinámico + toma de decisiones informada + participación ciudadana"  
**Para Comercio:** "Clientes nuevos + asociación con sostenibilidad + datos de efectividad"

### Tiempo de Creación
Estimado: 4-6 horas (es actualización del base, no creación desde cero)

## 2. PDF Institucional

### Descripción
El documento base convertido a PDF con formato profesional: logos, colores brand, tipografía coherente, diseño limpio.

### Audiencia
- Cualquiera que necesite una versión "pulida"
- Reuniones formales
- Gobierno, prensa, inversionistas

### Cuándo Usarlo
- En reuniones presenciales
- Para enviar a stakeholders
- En presentaciones donde quieres verse profesional
- Compartir públicamente

### Diferencias con el Base
| Aspecto | Base (MD) | PDF |
|---------|-----------|-----|
| Formato | Markdown (código) | Documento cerrado |
| Diseño | Ninguno (es crudo) | Diseño profesional |
| Editabilidad | Editable | No editable (seguro) |
| Apariencia | Técnica/borrador | Institucional/profesional |

### Estructura
Mismo contenido del base, pero:
- Portada profesional (logo, fecha, versión)
- Índice interactivo
- Headers y footers con branding
- Numeración de páginas
- Colores consistentes
- Tipografía profesional (no monospace)

### Herramientas para Generar
- Markdown to PDF: Pandoc + CSS personalizado
- Word: Exportar como PDF (si usas DOCX)
- Google Docs: Descargar como PDF
- Figma: Importar y diseñar (opción más profesional)

### Tiempo de Creación
Estimado: 3-4 horas (diseño básico), 8+ horas (diseño personalizado)

## 3. Documento DOCX Editable

### Descripción
El documento base en formato Word (.docx) para colaboración interna, edición y comentarios.

### Audiencia
- Tu equipo interno
- Consultores/asesores
- Cualquiera que use Word habitualmente

### Cuándo Usarlo
- Para colaboración (compartir versiones, track changes)
- Cuando alguien quiere editar o comentar
- En espacios corporativos donde Markdown no es estándar

### Diferencias con el Base
| Aspecto | Base (MD) | DOCX |
|---------|-----------|------|
| Formato | Markdown (texto plano) | Word (binario) |
| Track Changes | Git (para técnicos) | Word (fácil para todos) |
| Comentarios | Issues en GitHub | Comentarios Word |
| Estilo | Ninguno (renderiza) | Estilos Word (Heading 1, etc.) |

### Estructura
Mismo contenido del base, pero con:
- Estilos de Heading consistentes
- Indentación y espacios Word-native
- Tabla de contenidos automática
- Posibilidad de track changes
- Comentarios integrados

### Cómo Generarlo
1. Copiar contenido del .md
2. Pegar en Word
3. Aplicar estilos (Heading 1, Heading 2, Normal)
4. Ajustar espacios y formato
5. Generar tabla de contenidos automática

### Tiempo de Creación
Estimado: 2-3 horas (copy-paste + formato básico)

## 4. Pitch Deck (Presentación Visual)

### Descripción
Presentación de 10-15 diapositivas (PowerPoint/Keynote) que cuenta visualmente la historia de Arbu Games. Diseñada para ser presentada en vivo (15-20 minutos).

### Audiencia
- Comercios locales (pitch comercial)
- Gobiernos/municipios (pitch institucional)
- Inversionistas (pitch de inversión)

### Cuándo Usarlo
- Presentaciones en vivo
- Reuniones con potenciales aliados
- Eventos de pitch
- Cuando tienes 15-20 minutos de audiencia

### Diferencias con el Base
| Aspecto | Base | Pitch |
|---------|------|-------|
| Formato | Documento escrito | Presentación visual |
| Contenido | Exhaustivo | Síntesis (lo más importante) |
| Lectura | Para leer en 1 hora | Para escuchar en 15 min |
| Visuals | Ninguno | Imágenes, gráficos, iconos |
| Texto | Párrafos | 1-2 líneas por slide |

### Estructura Base (10 Diapositivas)
```
1. Portada (Logo + Título)
2. El Problema (Visualizado)
3. Por Qué es Importante (Datos/contexto)
4. La Solución (En 1 slide)
5. Cómo Funciona (Flujo visual)
6. Propuesta de Valor PARA ESTA AUDIENCIA
7. Mercado/Oportunidad (Números)
8. Diferencial/Por Qué Nosotros
9. Modelo Económico (Simple)
10. Llamada a Acción (Siguiente paso)
```

### Versiones por Audiencia
- **Para Comercios:** Enfasis en tráfico de clientes + ROI
- **Para Gobiernos:** Enfasis en datos + participación ciudadana
- **Para Inversionistas:** Enfasis en mercado + escalabilidad + financieros

### Herramientas
- PowerPoint / Keynote / Google Slides (básico)
- Figma (diseño profesional)
- Canva (rápido y con templates)

### Tiempo de Creación
Estimado: 6-10 horas (layout + contenido + diseño)

## 5. Business Model Canvas

### Descripción
Resumen visual de 1 página del modelo de negocio en 9 secciones (bloques): propuesta de valor, clientes, canales, ingresos, costos, recursos, actividades, asociados, relaciones.

### Audiencia
- Tu equipo interno
- Inversionistas
- Cualquiera que necesite "entender el negocio en 1 minuto"

### Cuándo Usarlo
- En conversaciones rápidas sobre el modelo
- Cuando alguien pregunta "¿Cómo ganan dinero?"
- En documentos ejecutivos
- Para identificar gaps en el modelo

### Diferencias con el Base
| Aspecto | Base | BMC |
|---------|------|-----|
| Formato | Prosa | Visual (9 bloques) |
| Extensión | 15 páginas | 1 página |
| Enfoque | Estrategia + contexto | Lógica de negocio |
| Lectura | 1 hora | 5 minutos |

### Los 9 Bloques del BMC
```
┌─────────────────┬──────────────────┬──────────────────┬──────────────┬──────────────────┐
│ ASOCIADOS CLAVE │ ACTIVIDADES CLAVE│ PROPUESTA DE     │ RELACIONES   │ SEGMENTOS        │
│                 │                  │ VALOR            │ CLIENTES     │ CLIENTES         │
├─────────────────┼──────────────────┼──────────────────┼──────────────┼──────────────────┤
│ Universidades   │ Mapeo + Validación│ Recompensas +    │ Community    │ Ciudadanos       │
│ Municipios      │ Gamificación      │ Datos útiles +   │ Support +    │ Comercios        │
│ ONGs            │ Billetera virtual │ Impacto local    │ Eventos      │ Gobiernos        │
│ Comercios       │ Admin dashboard   │                  │              │ Investigadores   │
└─────────────────┴──────────────────┴──────────────────┴──────────────┴──────────────────┘
│ RECURSOS CLAVE              │ CANALES                        │
├────────────────────────────────┼────────────────────────────────┤
│ Equipo dev                     │ App móvil (iOS/Android)        │
│ Plataforma (servidor)          │ Website                        │
│ Datos de árboles               │ Community/eventos              │
│ Partnerships                   │ Direct sales (comercios)       │
└────────────────────────────────┴────────────────────────────────┘
│ ESTRUCTURA DE COSTOS              │ FLUJOS DE INGRESOS                │
├─────────────────────────────────────┼──────────────────────────────────┤
│ Dev + salarios (60%)                │ Suscripción comercios (50%)       │
│ Infraestructura (15%)               │ Licencias municipios (25%)        │
│ Marketing (15%)                     │ Reportes para investigadores (15%)│
│ Recompensas a users (10%)           │ Patrocinios (10%)                │

```

### Cómo Generarlo
1. Crear en Excel / Figma / PowerPoint con tabla 3x3
2. Rellenar cada bloque con palabras clave
3. Conectar visualmente los bloques
4. Iterar (revisar relaciones)

### Herramientas
- Figma (diseño profesional)
- Miro / Mural (colaborativo)
- Canva (plantillas)
- Excel / PowerPoint (básico)

### Tiempo de Creación
Estimado: 2-3 horas

## 6. Roadmap de Producto

### Descripción
Timeline visual que muestra qué features/funcionalidades se construirán y cuándo. Divide el desarrollo en fases (usualmente trimestres o meses).

### Audiencia
- Tu equipo de desarrollo
- Inversionistas ("¿Cuándo sale X?")
- Socios estratégicos
- Posibles usuarios (early adopters)

### Cuándo Usarlo
- En planificación de sprints
- Cuando alguien pregunta "¿Cuándo?"
- En investor updates
- Para comunicar progreso

### Diferencias con el Base
| Aspecto | Base | Roadmap |
|---------|------|---------|
| Contenido | QUÉ hacemos y POR QUÉ | CUÁNDO lo hacemos |
| Perspectiva | Estratégica | Táctica/ejecución |
| Timeline | Sin tiempo | Con fechas |
| Detalles | Conceptos | Features específicas |

### Estructura Sugerida
```
ROADMAP DE ARBU GAMES

FASE 0: PRE-MVP (Mes 1-2)
├─ Semana 1-2: Setup infraestructura + inicio dev
├─ Semana 3-4: Core features (mapeo, foto, GPS)
└─ Semana 5-8: Validación comunitaria (1+3)

FASE 1: MVP SOFT LAUNCH (Mes 3)
├─ Billetera virtual + canjes
├─ Integración con comercios (5-10 pilotos)
└─ Testing con 50-100 beta users

FASE 2: MVP OPEN (Mes 4-6)
├─ Lanzamiento público Palermo
├─ Escalamiento a 500-800 usuarios
├─ Dashboard comercios
└─ Dashboard municipios (beta)

FASE 3: EXPANSIÓN (Mes 7-9)
├─ Segunda ciudad
├─ API para investigadores
├─ Reportes especializados
└─ Machine learning (especies)

FASE 4: ESCALA (Mes 10+)
├─ Tercera ciudad
├─ Expansion a otras regiones
├─ Nuevas funcionalidades
└─ Sustentabilidad financiera
```

### Tipos de Roadmap
- **High-level (para inversionistas):** Trimestres, pocos items
- **Mid-level (para equipo):** Meses, features agrupadas
- **Detailed (para devs):** Semanas, tasks específicas

### Herramientas
- Figma (visual)
- Jira Timeline (técnico)
- Notion (colaborativo)
- Asana (gestion)
- Timeline.gg (específico para roadmaps)

### Tiempo de Creación
Estimado: 3-4 horas (high-level), 8+ horas (detailed)

## 7. Plan de Validación del MVP

### Descripción
Documento que detalla **cómo** se van a probar las hipótesis críticas. Define métricas, método, success criteria y acciones based on results.

### Audiencia
- Tu equipo de producto/dev
- Inversionistas ("¿Cómo saben que funciona?")
- Socios que quieren entender el método

### Cuándo Usarlo
- Antes de lanzar MVP
- En reuniones donde preguntan "¿Cómo validan esto?"
- Para dirigir decisiones de QA/testing
- Como norte durante ejecución

### Diferencias con el Base
| Aspecto | Base | Plan Validación |
|---------|------|-----------------|
| Contenido | LISTA de hipótesis | CÓMO probar cada una |
| Enfoque | Identificar | Medir |
| Método | Narrativo | Operacional |
| Detalles | Conceptuales | Específicos (métricas) |

### Estructura Base
```
PLAN DE VALIDACIÓN DEL MVP

Introducción
- Duración MVP: 3 meses
- Ubicación: Buenos Aires - Palermo
- Usuarios meta: 500-800
- Hipótesis a validar: 8 principales

---

HIPÓTESIS 1: [NOMBRE]
Descripción: [La creencia]
Crítica: Sí/No (¿Es bloqueadora?)
Timeline: Validar en semana X

Métrica de éxito:
- KPI 1: [número] (ej: 70% de users mapean en D1)
- KPI 2: [número]

Método:
- Cómo mediremos: [describir]
- Con qué herramienta: [Mixpanel, etc.]
- Quién lo monitorea: [nombre]

Success Criteria:
- Si [métrica] ≥ [número] → VALIDADA
- Si [métrica] < [número] → REFUTADA

Acción si se refuta:
- Plan B: [cambio en estrategia]
- Timeline: [cuándo decidir]

---
[Repetir para cada hipótesis]

---

Resumen
- Si ≥7/8 hipótesis validadas → ESCALAR a otra ciudad
- Si <5/8 validadas → PIVOTEAR o PAUSE
```

### Ejemplo Concreto
```
HIPÓTESIS: Usuarios mapean árboles recurrentemente si hay recompensas

Métrica:
- % de users que mapean en D1: ≥70%
- Mapeos/usuario/semana (semana 3-8): ≥2

Método:
- Dashboard interno (Mixpanel)
- Monitoreo diario los primeros 7 días
- Semanalmente después

Success:
- D1 mapping ≥70% Y Mapeos/sem ≥2 → VALIDADA

Si falla (D1 <50% O Mapeos/sem <1):
- Plan B: Aumentar recompensa 2→4 puntos
- Simplificar flujo (quitar campos)
- Intensificar marketing
- Decidir en semana 2 si continuar o pivotar

Owner: Product Manager
```

### Herramientas para Medición
- Mixpanel, Amplitude (analytics)
- Google Analytics (web)
- Adjust, Firebase (mobile)
- Notion (tracking manual)

### Tiempo de Creación
Estimado: 4-6 horas (requiere pensar en métricas)

## 📝 Checklist para Cada Material

Antes de crear/actualizar cada material, verifica:

- [ ] ¿Audiencia clara? (¿para quién es?)
- [ ] ¿Propósito claro? (¿qué logra?)
- [ ] ¿Contenido alineado con el base?
- [ ] ¿Formato consistente con branding?
- [ ] ¿Accesible? (¿fácil de entender?)
- [ ] ¿Actualizado? (¿refleja versión actual del base?)
- [ ] ¿Compartible? (¿se puede mandar sin problemas?)

## 🔄 Cómo Mantenerlos Sincronizados

**Regla de oro:** El documento base es la **fuente única de verdad**.

Cuando cambies algo en el base:
1. Actualiza el base primero
2. Luego, actualiza los derivados en este orden:
   - PDF (re-exporta)
   - DOCX (copia-pega)
   - Pitch (resume cambios)
   - BMC (ajusta si aplica)
   - Roadmap (revisa timelines)
   - Validación (revisa hipótesis)
   - Estratégico (re-customiza)

**Frecuencia recomendada:** Revisar todos los derivados cada 2 semanas en fase MVP.

## 📌 Notas Finales

- **No es todo o nada:** No necesitas los 7 materiales desde el día 1. Crea según necesidad.
- **Reutilizable:** Muchos derivados se generan del mismo contenido del base.
- **Iterativo:** No son "hechos". Se actualizan conforme aprenden en MVP.
- **Específico por contexto:** El Pitch para comercios NO es el mismo que para gobiernos.
